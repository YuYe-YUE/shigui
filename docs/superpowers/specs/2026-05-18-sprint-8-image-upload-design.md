# Sprint 8 设计：单据图片上传

> 目标：为拾归系统补上“发布单据时上传图片”的能力，在不显著扩大系统复杂度的前提下，提高单据信息密度和找回效率。

## 1. 范围

Sprint 8 只实现以下能力：

- 小程序用户在发布 `LOST` / `FOUND` 单据时，最多上传 3 张图片
- 图片先上传到后端本地目录，再由发布接口引用这些图片 URL
- 首页卡片显示首图；详情页显示最多 3 张图
- 管理端审核详情页显示图片，但不支持补传、替换、删除

Sprint 8 明确不做：

- 对象存储（OSS/COS/S3）
- 管理端编辑图片
- 聊天图片消息
- 图片 AI 审核或图片识别
- 图片物理删除与垃圾回收策略

## 2. 设计原则

### 2.1 先拆“上传文件”和“发布单据”

图片上传与发布单据采用两步流程：

1. 小程序先调用文件上传接口，获得后端返回的图片 URL
2. 小程序再调用 `POST /api/posts`，把 `imageUrls` 一并提交

不采用一个 multipart 请求同时上传文件和表单，原因是：

- 当前后端和小程序发布链路已经基于 JSON 请求工作
- 拆成两步后，接口职责更清楚
- 失败重试更简单，前端体验也更容易做

### 2.2 用从表存图，不把 3 张图硬塞进主表

图片不存进 `lost_found_post` 的固定字段，而是新增 `post_image` 从表：

- 避免 `image_url_1 / image_url_2 / image_url_3` 这种僵硬结构
- 以后要扩到 5 张或更多时，不需要改主表
- 列表取首图、详情取全量都更自然

## 3. 数据库设计

新增表：`post_image`

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT AUTO_INCREMENT | 主键 |
| `post_id` | BIGINT NOT NULL | 关联 `lost_found_post.id` |
| `image_url` | VARCHAR(512) NOT NULL | 后端返回的可访问 URL |
| `sort_order` | INT DEFAULT 0 | 图片顺序，0/1/2 |
| `deleted` | TINYINT DEFAULT 0 | 逻辑删除 |
| `created_at` | DATETIME | 创建时间 |

索引建议：

- `INDEX idx_post_id (post_id)`
- `INDEX idx_post_sort (post_id, sort_order)`

### 3.1 单据与图片关系

- 一个 `lost_found_post` 对应 0 到 3 张 `post_image`
- 图片顺序由 `sort_order` 保证
- 单据删除时，图片记录不物理删除，只保留逻辑删除能力给后续 Sprint 使用

## 4. 后端设计

### 4.1 新增上传接口

新增接口：

- `POST /api/files/upload`

行为：

- 需要登录
- 接收单张图片文件
- 校验文件类型和大小
- 保存到后端本地目录
- 返回图片 URL

建议返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "url": "/uploads/posts/2026/05/18/uuid.jpg"
  }
}
```

### 4.2 本地存储方案

后端本地目录建议：

```text
backend/uploads/posts/YYYY/MM/DD/
```

命名规则建议：

- 使用随机 UUID 文件名
- 保留原始扩展名

这样可以避免：

- 文件名冲突
- 用户原始文件名包含特殊字符

### 4.3 静态访问

后端需要把 `/uploads/**` 映射为静态可访问资源，保证小程序和管理端都能直接显示图片。

### 4.4 发布接口扩展

`CreatePostRequest` 增加：

- `List<String> imageUrls`

发布校验规则：

- 最多 3 张
- 每个 URL 必须是后端本地上传返回的 URL
- 不接受任意外链

发布成功后：

1. 保存 `lost_found_post`
2. 遍历 `imageUrls`
3. 按顺序写入 `post_image`

### 4.5 查询接口扩展

以下接口需要返回图片：

- `GET /api/posts`
- `GET /api/posts/{id}`
- `GET /api/admin/posts`
- `GET /api/admin/posts/{id}`

返回规则：

- 列表接口：返回 `coverImageUrl`（首图）
- 详情接口：返回 `imageUrls`

这样列表更轻，详情更完整。

## 5. 小程序设计

### 5.1 发布页

`pages/publish-form/publish-form` 增加图片上传区，能力包括：

- 选择图片
- 最多 3 张
- 缩略图预览
- 删除已选图片
- 上传中状态提示

流程：

1. 用户选择本地图片
2. 前端调用 `wx.uploadFile` 逐张上传
3. 上传成功后保存返回的 URL
4. 提交单据时把 `imageUrls` 一起发给 `POST /api/posts`

### 5.2 首页卡片

`post-card` 改为：

- 有 `coverImageUrl` 时显示真实图片
- 没有时继续使用当前占位图

### 5.3 详情页

详情页显示最多 3 张图，优先采用简单可用方案：

- 顶部横向图片列表或轮播
- 点击可预览原图

Sprint 8 不做复杂图片编辑交互。

## 6. 管理端设计

管理端只做只读展示：

- 审核列表里可显示首图缩略图
- 审核详情里可显示全部图片

Sprint 8 不做：

- 补传
- 替换
- 删除图片

原因是管理端改图会立即引入：

- 权限边界问题
- 审计问题
- 原图替换与前端缓存问题

这些都不适合在本 Sprint 展开。

## 7. 校验与限制

### 7.1 数量限制

- 每条单据最多 3 张图

### 7.2 格式限制

允许：

- `jpg`
- `jpeg`
- `png`
- `webp`

不允许：

- gif
- pdf
- 任意非图片文件

### 7.3 大小限制

建议：

- 单张图片最大 5MB

### 7.4 安全边界

- 后端必须校验 MIME / 扩展名，不信任前端声明
- 发布接口只接受本系统上传返回的 URL
- 不允许前端直接写第三方 URL

## 8. 测试设计

### 8.1 后端

至少覆盖：

- 上传接口成功
- 上传接口拒绝非图片
- 上传接口拒绝超大文件
- 发布接口允许 0 到 3 张图
- 发布接口拒绝超过 3 张图
- 列表接口返回 `coverImageUrl`
- 详情接口返回 `imageUrls`

### 8.2 小程序

至少覆盖：

- 选择图片后本地预览状态正确
- 超过 3 张时被拦截
- 删除图片后可继续补选
- 上传成功后提交 payload 带 `imageUrls`

### 8.3 管理端

至少覆盖：

- 列表页有首图时正常显示
- 详情页能显示图片数组
- `npm run build` 通过

## 9. 验收标准

Sprint 8 完成后，应满足：

1. 小程序发布单据时可上传最多 3 张图
2. 图片上传成功后能随单据一起保存
3. 首页卡片可显示首图
4. 详情页可查看全部图片
5. 管理端审核可查看图片
6. 没有图片的旧单据仍能正常展示，占位图不受影响

## 10. 风险与后续

### 10.1 当前接受的妥协

- 图片存在本地，不适合多实例部署
- 删除单据不清理物理文件，后续需要补清理策略

### 10.2 后续可以接的增强项

- 对象存储迁移
- 图片压缩与缩略图
- 管理端图片审核
- 聊天图片消息
- 图片 AI 审核

---

这是一个典型的 MVP 增强 Sprint：它明显提升产品可用性，但仍然保持实现边界清晰，不把系统一下拉进“媒体平台”复杂度。
