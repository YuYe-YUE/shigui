## 1. 引言

### 1.1 编写目的

本文档为"拾归"（Shi-Gui）校园失物招领系统的软件详细设计说明书。在《软件体系结构设计》确定总体框架后，本文档将每个模块的接口、数据结构、算法和业务规则细化到可直接编码的程度。

读者包括：开发团队（编码依据）、测试团队（测试用例设计依据）、维护人员（系统理解依据）。

### 1.2 范围

涵盖系统的三个子系统：

| 子系统 | 技术栈 | 交付物 |
|--------|--------|--------|
| 后端服务 | Spring Boot 3.5.14 + MyBatis-Plus 3.5.16 + Sa-Token 1.45.0, Java 21, Maven | REST API, 81 个源文件 |
| 管理端 Web | Vue 3 + Vite + TypeScript + Element Plus + Pinia + Vue Router | 6 页 SPA |
| 用户端小程序 | 原生微信小程序框架 | 11 页面 + 1 组件 |

### 1.3 术语与缩写

| 术语 | 说明 |
|------|------|
| LOST | 寻物单——失主发布，描述丢失的物品 |
| FOUND | 招领单——拾捡者发布，描述捡到的物品 |
| 私密特征 (privateFeature) | 只有物品主人知道的特征描述，用于认领核验 |
| Sa-Token | 轻量级 Java 权限认证框架 |
| `@TableLogic` | MyBatis-Plus 逻辑删除注解，自动在查询中添加 `deleted=0` |
| MATCHING | 单据已通过审核、进入匹配池等待配对 |
| IDOR | Insecure Direct Object Reference，不安全对象直接引用（越权漏洞） |

### 2.2 技术选型理由

| 技术 | 选型理由 |
|------|---------|
| Spring Boot 3.5 | 成熟的企业级框架，自动配置、starter 生态完整 |
| MyBatis-Plus 3.5 | Lambda 类型安全查询，代码生成器，分页插件，`@TableLogic` 逻辑删除 |
| Sa-Token 1.45 | 比 Spring Security 轻量，注解式鉴权，支持自定义 `loginId` 策略（管理员 = 用户 ID + 偏移量） |
| MySQL 8.0 | 关系型数据，事务 ACID 保证，`utf8mb4` 支持 Emoji |
| Vue 3 + Element Plus | 组件库丰富，TypeScript 类型安全，Vite 秒级热更新 |
| 原生微信小程序 | 无额外框架学习成本，直接调用微信能力（登录、图片预览） |
| DeepSeek v4-flash | 极高性价比，语义理解能力强，支持 JSON 结构化输出 |
| JUnit 5 + Mockito | Spring Boot 标准测试栈，MockMvc 模拟 HTTP 请求 |
| Maven Wrapper | 无需全局安装 Maven，`mvnw` 自动下载匹配版本 |

### 2.3 分层架构原则

```
Controller ──→ Service 接口 ──→ ServiceImpl ──→ Mapper ──→ DB
   │               │                  │              │
   │         接口定义              @Transactional    MyBatis-Plus
   │         (可 Mock)            业务编排           BaseMapper
   │                                                 自动生成 CRUD
   └── 入参校验、Result 封装、权限注解
```

**铁律**：
- Controller 不包含业务逻辑，只做参数校验和 Result 封装
- Service 是事务边界，`@Transactional` 只加在需要原子操作的方法上
- Mapper 只定义数据访问，复杂查询用 `LambdaQueryWrapper`，禁止字符串拼接 SQL
- 跨模块调用只能通过 Service 接口，禁止直接调用 Mapper

## 3. 数据库详细设计

### 3.1 数据库环境

- **数据库名**: `shi_gui`
- **引擎**: InnoDB（全部 11 张表）
- **字符集**: `utf8mb4`, 排序规则 `utf8mb4_unicode_ci`
- **逻辑删除**: 统一使用 `deleted TINYINT DEFAULT 0`（0=正常, 1=已删）
- **时间戳**: `created_at DATETIME DEFAULT CURRENT_TIMESTAMP`, `updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`

### 3.2 实体关系图

```
app_user ──1:N──→ lost_found_post ──1:N──→ post_image
   │                    │
   │                    ├──1:N──→ audit_record ──N:1── admin_user
   │                    │
   │                    ├──1:N──→ claim_record
   │                    │              └── 唯一约束: 每个 post 最多一个活跃认领
   │                    │
   │                    ├──1:N──→ match_record (LOST ↔ FOUND 对)
   │                    │
   │                    └──1:N──→ chat_session
   │                                   └──1:N──→ chat_message
   │
   ├──1:N──→ notification
   │
   └── 独立的 system_config (键值对配置表)
```

#### 3.3.2 admin_user — 后台管理员

| 字段 | 类型 | 约束 | 默认值 | 说明 |
|------|------|------|--------|------|
| id | BIGINT | PK, AUTO_INCREMENT | — | 管理员唯一标识 |
| username | VARCHAR(64) | NOT NULL, UNIQUE | — | 登录用户名 |
| password_hash | VARCHAR(256) | NOT NULL | — | 密码哈希，格式 `salt:SHA256(salt+SHA256(password))` |
| deleted | TINYINT | — | 0 | 逻辑删除 |
| created_at | DATETIME | — | NOW() | 创建时间 |
| updated_at | DATETIME | ON UPDATE | NOW() | 更新时间 |

**密码存储方案**:

```
前端:  hashed = SHA256(password)          // 发送 hashed，不是明文
后端:  stored = salt + ":" + SHA256(salt + hashed)
验证:  SHA256(salt + 收到的hashed) == hash
```

双层哈希确保即使数据库泄露，攻击者无法从 hash 反推出原始密码直接登录。

**管理员 ID 映射**: Sa-Token `loginId = adminId + 10_000_000`。`requireAdmin()` 检查 `loginId >= 10_000_000`，普通用户 token 访问管理接口返回 403。

### 3.4 状态流转设计

#### 3.4.1 单据状态 (lost_found_post.status)

```
用户发布
    │
    ▼
PENDING_AUDIT ──── 管理员删除 ────→ deleted=1（终端状态）
    │
    │ 管理员审核通过
    ▼
MATCHING ──── 有人认领 ────→ CLAIMING
    │                              │
    │ 管理员删除                    │ 认领完成
    ▼                              ▼
deleted=1                    RETURNING ──→ COMPLETED
（终端状态）                                  （终端状态）
```

#### 3.4.2 认领状态 (claim_record.status)

```
用户提交认领
    │
    ▼
PENDING_AI_REVIEW ──── AI 判定 ──┬──→ VERIFIED（高置信度匹配）
                                 ├──→ REJECTED（明确不匹配）
                                 └──→ PENDING_ADMIN_REVIEW（不确定，转人工）
                                          │

```

## 4. API 接口详细设计

### 4.1 接口规范

**统一响应体**:

```json
{
  "code": 200,        // 业务状态码: 200=成功, 400=参数错误, 401=未登录, 403=无权限, 404=不存在, 500=服务器错误
  "message": "success",  // 提示信息
  "data": {}             // 响应数据（可为 null）
}
```

**鉴权方式**: 请求头 `satoken: <token_value>`。登录成功后由后端返回 token，前端存入本地存储。

**分页规范**: 查询参数 `?page=1&size=10`，响应 `{records: [], total: N}`。

### 4.2 鉴权拦截配置

Sa-Token 路由拦截规则：

```
拦截: /api/**
排除: /api/user/wx-login     → 微信登录（公开）
排除: /api/admin/login        → 管理员登录（公开）
排除: /api/posts              → 单据公开列表
排除: /api/posts/{id}         → 单据详情（Controller 内手动判断）
排除: /api/posts/map          → 地图点位
```

`/api/posts/mine` 等需要登录的接口在 Controller 方法内手动调用 `StpUtil.checkLogin()`。

### 4.3 用户认证接口

#### POST /api/user/wx-login — 微信登录

```
Body:   { "openid": "dev_<wx.login() code>" }
响应:   { "code": 200, "data": "<satoken>" }
```

开发环境使用 `dev_` 前缀模拟微信 OpenID。首次登录自动注册用户。

#### GET /api/user/me — 当前用户信息

```
Header: satoken: <token>
响应:   { "code": 200, "data": { "id": 1, "nickname": "...", "status": "NORMAL", ... } }
```

### 4.4 管理员认证接口

#### POST /api/admin/login — 管理员登录

```
Body:   { "username": "admin", "password": "<SHA256(password)>" }
响应:   { "code": 200, "data": "<satoken>" }
错误:   400 — "用户名和密码不能为空" / "用户名或密码错误"
```

**安全要点**: password 字段为前端 SHA-256 哈希后的值，非明文。后端再做 `SHA256(salt + hashed)` 与数据库比对。

### 4.5 单据接口（公开 + 需登录）

#### POST /api/posts — 发布单据 (需登录)

```
Body: {
  "postType": "LOST | FOUND",
  "title": "丢失校园卡",
  "itemName": "校园卡",
  "itemCategory": "校园卡",
  "description": "昨天在逸夫楼丢失...",
  "privateFeature": "卡背面贴有熊猫贴纸",   // 非必填
  "campusArea": "南校园",
  "locationName": "逸夫楼 301",
  "longitude": 113.1234567,
  "latitude": 23.1234567,
  "storageLocation": "逸夫楼保安亭",        // 仅 FOUND
  "eventTime": "2026-05-18T10:00:00",
  "imageUrls": ["/uploads/posts/2026/05/18/uuid.jpg"]  // 最多 3 个
}
```

**校验规则**:
- postType 必须是 `LOST` 或 `FOUND`，否则 400
- title / itemName / itemCategory / campusArea / locationName / eventTime 必填
- imageUrls 最多 3 个，每个 URL 必须以 `/uploads/posts/` 开头
- 图片 URL 需通过 `FileStorageService.isStoredPostImage()` 验证文件确实存在
- BANNED 用户禁止发布（返回 400）

#### GET /api/posts — 公开列表

```
参数:   ?page=1&size=10&postType=LOST|FOUND&itemCategory=校园卡&campusArea=南校园&keyword=校园卡
响应:   { "code": 200, "data": { "records": [...], "total": N } }
```

**过滤规则**: 仅返回 `status=MATCHING AND deleted=0` 的单据。`privateFeature` 字段不会出现在响应中。

#### GET /api/posts/{id} — 单据详情

```
响应: 200, PostResponse (含 imageUrls, coverImageUrl, canChat)
      404 — 单据不存在
      400 — 单据不可见（非 MATCHING 状态且不是本人/已认领通过者）
```

**可见性规则**（三层）:
1. 发布者本人 → 始终可见
2. 已通过认领的用户 → 始终可见
3. 其他 → 仅 MATCHING 状态可见

**canChat 字段**:
- FOUND 招领单：仅认领审核通过 (VERIFIED) 的认领者可以聊天
- LOST 寻物单：非发布者可直接联系
- 发布者本人：canChat = false（不能和自己聊天）

#### GET /api/posts/mine — 我的记录 (需登录)

```
参数:   ?page=1&size=10&postType=LOST|FOUND
响应:   { "records": [...], "total": N }
```

#### GET /api/posts/map — 地图点位

```
响应: [ { "id": 1, "itemName": "校园卡", "longitude": 113.12, "latitude": 23.12, ... } ]
```

仅返回 `postType=FOUND AND status=MATCHING AND deleted=0 AND longitude IS NOT NULL AND latitude IS NOT NULL` 的单据。

### 4.6 匹配与通知接口

#### GET /api/matches/mine — 我的匹配 (需登录)

```
参数:   ?page=1&size=10
响应:   { "records": [ { "id": 1, "lostPostId": 10, "foundPostId": 20, "score": 0.88, "reason": "..." } ], "total": N }
```

#### GET /api/notifications — 通知列表 (需登录)

```
参数:   ?page=1&size=10
响应:   { "records": [ { "id": 1, "title": "匹配通知", "content": "...", "isRead": 0 } ], "total": N }
```

### 4.7 认领接口

#### POST /api/claims — 发起认领 (需登录)

```
Body:   { "postId": 1, "privateFeatureAnswer": "背面有熊猫贴纸" }
响应:   200 — 成功
        400 — 用户被封禁 / 不能认领自己的单据 / 已有活跃认领
```

**流程**: 提交后异步调用 AI 预审。AI 返回高置信度匹配 → 自动通过 (VERIFIED)；明确不匹配 → 自动拒绝 (REJECTED)；不确定 → 转人工 (PENDING_ADMIN_REVIEW)。

#### GET /api/claims/mine — 我的认领 (需登录)

```
参数:   ?page=1&size=10
响应:   { "records": [...], "total": N }
```

#### GET /api/claims/{id} — 认领详情 (需登录)

**鉴权**: 校验 `claimantUserId == 当前用户`，防止 IDOR 越权查看他人认领。

#### PUT /api/claims/{id}/confirm-receive — 确认收到 (需登录)

**鉴权**: 校验认领所有权。将状态更新为 COMPLETED。

### 4.8 聊天接口

#### POST /api/chat/sessions — 创建/复用会话 (需登录)

```
Body:   { "postId": 1 }
响应:   { "code": 200, "data": { "id": 1, "postId": 1, ... } }
```

**安全校验**:
- BANNED 用户禁止创建
- 不能与自己发布的单据聊天
- FOUND 招领单：必须已有 VERIFIED 状态的认领记录
- 同一 post + 同一对用户已存在会话时，返回已有会话

#### GET /api/chat/sessions/{id}/messages — 消息列表 (需登录)

**鉴权**: `requireParticipantSession()` 校验当前用户是会话的 `lostUserId` 或 `foundUserId`。

#### POST /api/chat/sessions/{id}/messages — 发消息 (需登录)

```
Body:   { "content": "你好，请问校园卡还在吗？" }
```

**鉴权**: 同上。

### 4.9 文件上传接口

#### POST /api/files/upload — 上传图片 (需登录)

```
Content-Type: multipart/form-data
参数:   file (图片文件)
响应:   { "code": 200, "data": { "url": "/uploads/posts/2026/05/19/uuid.jpg" } }
```

**校验**:
- 格式：仅 JPG / PNG / WebP（通过文件魔数校验，非扩展名）
- 大小：最大 5MB
- 路径：按日期分目录 `uploads/posts/YYYY/MM/DD/UUID.ext`
- 安全：防止路径穿越（`..`、`/` 等字符过滤）

### 4.11 错误码汇总

| code | HTTP Status | 含义 | 触发条件 |
|------|-------------|------|---------|
| 200 | 200 | 成功 | — |
| 400 | 200/400 | 参数错误 | 必填缺失、格式非法、业务规则违反 |
| 401 | 401 | 未登录 | token 缺失/过期，Sa-Token `NotLoginException` |
| 403 | 403 | 权限不足 | 普通用户访问管理接口 |
| 404 | 200/404 | 不存在 | 单据/认领/会话等资源不存在 |
| 405 | 405 | 方法不支持 | GET 访问 POST 接口等 |
| 500 | 500 | 服务器错误 | 未预料的异常（SQL 异常、NPE 等，由兜底 Handler 捕获） |

**注意**: 业务错误（如 400）在 HTTP 层面仍返回 200，通过 `code` 字段区分。仅认证/授权/服务器错误使用对应的 HTTP 状态码，便于 Nginx 日志和监控。

#### 5.2.1 LostFoundPostServiceImpl — 单据服务 (333 行)

| 方法 | 事务 | 说明 |

| `publish(userId, request)` | `@Transactional` | 封禁校验 → 参数校验 → 保存单据 → 保存图片关联 |
| `getDetail(postId, currentUserId)` | 无 | 三层可见性判断 → 加载图片 → 设置 canChat |

**关键实现细节**:

#### 5.2.2 ClaimRecordServiceImpl — 认领服务 (357 行)

**公开方法**:

| 方法 | 事务 | 说明 |
|------|------|------|
| `createClaim(userId, postId, answer)` | `@Transactional` | BANNED 校验 → 不能认领自己的 → 活跃认领唯一检查 → 异步 AI 预审 → 保存 → 更新 post 状态为 CLAIMING |
| `approveByClaimAi(claim)` | 无 | AI 回调：高置信度 MATCH → VERIFIED，NOT_MATCH → REJECTED |
| `approveByAdmin(id)` | 无 | 管理员审核通过 → VERIFIED |
| `rejectByAdmin(id, reason)` | 无 | 管理员拒绝 → REJECTED |

**AI 异步调用**: `@Async` 注解，不阻塞用户请求。AI 失败时默认转 PENDING_ADMIN_REVIEW（人工审核兜底）。

#### 5.2.3 MatchRecordServiceImpl — 匹配服务 (271 行)

**匹配触发**: 单据审核通过时，`AdminPostService.approvePost()` 在事务提交后调用 `generateMatchesForPost()`。

**匹配算法（两阶段）**:

```
阶段 1 — 规则预筛（数据库查询）:
  - 同校区 (campus_area 匹配)
  - 同品类 (item_category 匹配)
  - 时间差 < 30 天 (event_time 差值)
  - 关键词匹配 (title 文本交集)

阶段 2 — AI 语义排序:
  - 将预筛结果发送给 DeepSeek v4-flash
  - AI 对每个候选对打分 (0-1)
  - 输出结构化 JSON: [{postId, score, reason}]
  - 阈值过滤 (score >= 0.7)
  - AI 失败时规则兜底（分数 = 规则匹配度）
```

**去重**: 每次匹配前检查 `match_record` 唯一约束 `uk_post_pair`，已有记录不重复生成。

### 5.3 异常处理设计

`GlobalExceptionHandler` 使用 6 个 `@ExceptionHandler` 统一转换：

```
NotLoginException           → 401 "未登录或登录已过期"
NotPermissionException       → 403 "权限不足"
IllegalArgumentException     → 400 具体业务错误
HttpRequestMethodNotSupportedException → 405
NoResourceFoundException    → 404 "资源不存在"
Exception (兜底)             → 500 "服务器内部错误"（不暴露堆栈）
```

**Service 层异常规范**: 所有业务校验失败统一抛 `IllegalArgumentException("中文错误描述")`，由全局 Handler 捕获并返回给前端。

### 5.4 AI 集成设计

**配置管理** (`AiMatchProperties` / `AiClaimReviewProperties`):


### 5.5 文件存储设计

`FileStorageService` 接口抽象文件操作，当前实现 `LocalFileStorageServiceImpl` (130 行)：

**存储路径**: `uploads/posts/YYYY/MM/DD/UUID.ext`（相对于工作目录）

**安全措施**:
- 文件类型校验：读取文件头魔数（Magic Number），非扩展名判断

  - PNG: `89 50 4E 47`

- 文件大小限制: 5MB
- 文件名生成: `UUID.randomUUID() + 原始扩展名`

## 7. 安全设计

### 7.1 认证机制

| 层级 | 机制 | 说明 |
|------|------|------|
| 小程序 | 微信 `wx.login()` 获取 code → 后端置换 token | 开发环境 `dev_` 前缀模拟 |
| 管理端 | 用户名 + SHA-256 哈希密码 | 前端先哈希（防明文传输），后端再双层哈希存储 |
| 鉴权 | Sa-Token 请求头 `satoken` | 每次请求携带，后端 `StpUtil.checkLogin()` 校验 |

### 7.2 授权机制

| 接口类别 | 鉴权方式 | 失败响应 |
|---------|---------|---------|
| 公开接口 | 无需鉴权 | — |
| 需登录接口 | `StpUtil.checkLogin()` | 401 |
| 管理端接口 | `requireAdmin()` = `loginId >= 10_000_000` | 403 |
| 用户私有接口 | Service 层校验所有权 | 400（防止 IDOR） |

**管理员 ID 设计原理**: `loginId = adminId + 10_000_000`。由于小程序用户 ID 从 1 开始自增，不超过 10M，管理员可通过 `loginId >= 10_000_000` 简单判断，无需维护额外的角色表或 RBAC 模型。

### 7.3 数据安全

**密码存储**:
```
数据库:  salt:SHA256(salt + SHA256(password))
传输层:  SHA256(password)（非明文）
```

即使数据库泄露，攻击者无法：
1. 从 hash 反推原始密码（SHA-256 单向性）
2. 用泄露的 hash 直接登录（后端会再哈希一次）

**防水平越权 (IDOR)**:
- 认领详情 `GET /api/claims/{id}` → 校验 `claimantUserId == 当前用户`
- 确认收到 `PUT /api/claims/{id}/confirm-receive` → 同上
- 聊天消息 `GET /api/chat/sessions/{id}/messages` → 校验用户属于会话参与者
- 发消息 `POST /api/chat/sessions/{id}/messages` → 同上
- 发布者不能与自己发布的单据聊天（`ChatServiceImpl.createSession` 校验）

**输入校验**:
- 图片上传：魔数校验文件类型 + 5MB 上限 + 防路径穿越
- 发布单据：`imageUrls` 最多 3 个 + 必须以 `/uploads/posts/` 开头 + 文件存在性验证
- 参数类型：Controller 层 Spring 自动类型转换 + 业务层二次校验

**敏感信息脱敏**:
- 公开列表不返回 `privateFeature`
- 聊天消息不暴露用户真实身份，只显示 LOST/FOUND 角色

### 7.4 网络与部署安全

- CORS 配置：`CorsConfig` 精确控制允许的来源和方法
- HTTPS：生产环境部署时 Nginx 配置 SSL 证书
- 管理端 Vite 代理：开发环境 `/api` 代理到本地 8080，避免跨域问题

## 8. 业务流程详细设计

### 8.1 用户注册与登录

```
───────                         ──────                         ──────
wx.login() → code
    │

    │   { openid: "dev_"+code }     │
    │                               ├─ 存在 → 返回 token
    │                               └─ 不存在 → INSERT 新用户 → 返回 token
    │                                             │
    ◄── { code: 200, data: token } ←─────────────┘
    │
    ├─ 存入 Storage + globalData
    └─ GET /api/user/me ──→ 返回完整用户信息
```

### 8.2 单据发布与审核

```
用户发布
    │
    ▼
POST /api/posts ──→ LostFoundPostServiceImpl.publish()
    │                   │
    │                   ├─ BANNED 用户校验
    │                   ├─ 字段非空校验
    │                   ├─ 图片 URL 格式 + 文件存在性校验
    │                   ├─ INSERT lost_found_post (status=PENDING_AUDIT)
    │                   └─ INSERT post_image × N
    │
    ▼
管理员在"内容审核"页看到待审核单据
    │
    ├─ 点击"详情" → ElDialog 展示完整信息 + 图片
    │
    ├─ 点击"通过" ──→ POST /api/admin/posts/{id}/approve
    │                   │
    │                   └─ AdminPostService.approvePost()
    │                         │ (@Transactional)
    │                         ├─ UPDATE status = MATCHING
    │                         ├─ INSERT audit_record (action=APPROVE)
    │                         └─ @Transactional 提交后:
    │                               └─ MatchRecordService.generateMatchesForPost()
    │                                     ├─ 规则预筛候选
    │                                     ├─ AI 语义排序
    │                                     ├─ INSERT match_record × N
    │                                     └─ INSERT notification × N
    │
    └─ 点击"删除" → 二次确认 → 填写原因 → DELETE /api/admin/posts/{id}
                        │
                        └─ AdminPostService.deletePost()
                              └─ UPDATE deleted = 1
                              └─ INSERT audit_record (action=DELETE)
```

### 8.3 智能匹配详细流程

```
新单据审核通过（以 LOST 为例）
    │
    ▼
MatchRecordService.generateMatchesForPost(lostPost)
    │
    ├── 1. 规则预筛（数据库查询）
    │      SELECT FROM lost_found_post
    │      WHERE post_type = 'FOUND'
    │        AND status = 'MATCHING'
    │        AND deleted = 0
    │        AND campus_area = lostPost.campusArea
    │        AND item_category = lostPost.itemCategory
    │        AND ABS(DATEDIFF(event_time, lostPost.eventTime)) < 30
    │      ORDER BY published_at DESC
    │      LIMIT 20
    │
    ├── 2. AI 语义排序
    │      POST https://api.deepseek.com/v1/chat/completions
    │      Body: {
    │        model: "deepseek-v4-flash",
    │        messages: [{
    │          role: "system",
    │          content: "你是校园失物招领匹配专家。根据物品特征评估匹配度..."
    │        }, {
    │          role: "user",
    │          content: "LOST: {title, itemName, description} | FOUND 候选: [{...}, ...]"
    │        }],
    │        response_format: { type: "json_object" }
    │      }
    │
    │      AI 响应: { matches: [{postId, score, reason}, ...] }
    │
    ├── 3. 去重去阈值
    │      WHERE score >= 0.7  (AiMatchProperties.threshold)
    │      AND pair NOT IN match_record  (UNIQUE constraint)
    │      ORDER BY score DESC LIMIT 10
    │
    └── 4. 持久化 + 通知
           INSERT INTO match_record (lost_post_id, found_post_id, score, reason)
           INSERT INTO notification (user_id, type='MATCH', ...)
             → 失主 + 拾捡者双方各一条
```

**AI 失败处理**: 如果 DeepSeek API 调用异常（超时、网络错误、返回格式异常），跳至规则兜底——按规则匹配数量计分，生成匹配记录并标记。

### 8.4 认领详细流程

```
认领者点击"申请认领"（仅 FOUND 招领单显示）
    │
    ▼
wx.showModal({ editable: true }) → 输入私密特征答案
    │
    ▼
POST /api/claims { postId, privateFeatureAnswer }
    │
    ▼
ClaimRecordServiceImpl.createClaim()
    │  @Transactional
    ├─ BANNED 用户校验
    ├─ 不能认领自己的单据
    ├─ 活跃认领检查 (uk_active_claim_post UNIQUE 约束)
    ├─ INSERT claim_record (status=PENDING_AI_REVIEW)
    │
    └─ @Async → AI 预审 (OpenAiCompatibleClaimReviewClient)
                    │
                    ├─ 高置信度匹配 → UPDATE status=VERIFIED, verified_at=NOW()
                    ├─ 明确不匹配   → UPDATE status=REJECTED
                    └─ 不确定       → UPDATE status=PENDING_ADMIN_REVIEW

管理员审核（PENDING_ADMIN_REVIEW 时）
    │
    ├─ 通过 → PUT /api/admin/claims/{id}/approve → VERIFIED
    └─ 拒绝 → PUT /api/admin/claims/{id}/reject  → REJECTED

认领通过后 (VERIFIED)
    │
    ├─ 详情页 canChat=true → "联系对方" 按钮出现
    └─ 确认收到 → PUT /api/claims/{id}/confirm-receive → COMPLETED
```

### 8.5 匿名聊天详细流程

```
前提：用户看到 canChat=true（LOST 无关、FOUND 需认领通过）
    │
    ▼
点击"联系对方" → 跳转 /pages/chat/chat?postId=X
    │
    ▼
POST /api/chat/sessions { postId }
    │
    ▼
ChatServiceImpl.createSession(userId, postId)
    ├─ BANNED 用户校验
    ├─ 不能和自己聊天
    ├─ FOUND 单据：检查 claim_record (status=VERIFIED AND claimantUserId=userId)
    ├─ 查询已存在会话 (same post + same users) → 有则复用
    ├─ 确定角色：lostUserId = post.userId, foundUserId = post 的发布者
    ├─ INSERT chat_session
    └─ 返回 session

获取消息
    │

发送消息
    │
    ▼
POST /api/chat/sessions/{id}/messages { content }
    ├─ requireParticipantSession()
    └─ INSERT chat_message
```
