## 2. 测试策略

### 2.1 测试金字塔

```
              ╱──────╲
             ╱  E2E   ╲        1 个端到端测试（全链路）
            ╱──────────╲      ─────────────────────────
           ╱ Controller ╲     54 个集成测试（MockMvc）
          ╱──────────────╲   ─────────────────────────
         ╱    Service     ╲   46 个单元测试（Mockito）
        ╱──────────────────╲ ─────────────────────────
       ╱   Schema / Infra   ╲ 10 个合约/基础设施测试
      ╱──────────────────────╲─────────────────────────
```

### 2.2 各层测试策略

| 层级 | 策略 | 工具 | 数据库 |
|------|------|------|--------|
| Service 单元 | Mock 所有 Mapper，验证纯业务逻辑 | Mockito | 无 |
| Controller 集成 | Mock 所有 Service，验证 HTTP 层行为 | MockMvc + `@MockitoBean` | 无 |
| 端到端 | 不 Mock，走完整调用链 | `@SpringBootTest` + MockMvc | 真实 MySQL |
| Schema 合约 | 对比实体类与 DDL | JUnit 5 | H2 内存库 |
| AI 客户端 | 条件执行（需环境变量） | JUnit 5 | 无 |

### 2.3 隔离原则

- **Service 测试**: 每个测试方法独立，`@BeforeEach` 构造被测对象并注入 mock Mapper。
- **Controller 测试**: `@MockitoBean` 为每个测试类注入 mock Service，`@SpringBootTest` 加载完整 Web 上下文。
- **SaToken 鉴权隔离**: 通过 `SaTokenContextJakartaServletUtil.setContext(request, response)` 手动设置上下文，使用 `StpUtil.login()` 模拟不同角色登录（`1L` = 普通用户，`1L + 10_000_000L` = 管理员）。
- **端到端测试**: `@Transactional + @Rollback`，测试完毕后自动回滚，不污染数据库。

## 6. 端到端集成测试

### 6.1 EndToEndMatchingTest (1 测试)

**测试方法**: `approveLostPost_generatesMatchAndNotifications`  
**类型**: `@SpringBootTest + @AutoConfigureMockMvc + @Transactional + @Rollback`  
**数据库**: 真实 MySQL（需要 seed_data 已执行）

**测试流程**:

```
Step 1: User A 微信登录 (openid=e2e_lost_user)
Step 2: User B 微信登录 (openid=e2e_found_user)
Step 3: Admin 登录 (admin / SHA256(admin123))
Step 4: User B 发布 FOUND 招领单 → admin 审核通过
Step 5: User A 发布 LOST 寻物单 → admin 审核通过
       → @Transactional 提交后触发 MatchRecordService.generateMatchesForPost()
Step 6: 验证 User A 的匹配记录非空（GET /api/matches/mine）
Step 7: 验证 User A 的通知非空（GET /api/notifications）
```

**覆盖链路**: 微信登录 → 发布单据 → 管理员审核 → AI 匹配触发 → 匹配记录写入 → 通知写入

**隔离**: `@Transactional + @Rollback`，测试结束后自动回滚，不污染数据库。
